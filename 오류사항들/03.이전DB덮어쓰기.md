# 오류 보고서: 구버전 DB 누적 및 RAG 검색 범위(k) 부족에 따른 검색 실패(누락) 

## 🚨 문제 상황 (Problem)
AI 챗봇에게 "1구역 가스터빈 1호기(T-101) 온도가 너무 높아"라고 질문하자, 분명히 `turbine_manual.txt` 매뉴얼 안에 해당 설비 내용이 존재함에도 불구하고:
**"죄송합니다. 다시 검색해 보았지만 'T-101'에 대한 매뉴얼은 찾을 수 없습니다. 혹시 G-101 설비를 찾으시는 것이 맞으실까요?"** 라며 엉뚱한 설비를 제시하는 현상 발생.

## 🔍 원인 분석 (Cause)
사용자님의 지적(의문)이 아주 날카롭고 정확합니다! *"원래 있던 데이터인데 덮어쓰기해도 가져와야 하는 거 아닌가요?"* -> **네, 가져오는 것이 맞습니다. 하지만 문제는 'AI의 시야(검색 개수, k)'가 너무 좁았다는 데 있습니다.**

1. **근본 역전 현상 (Vector 유사도):**
   우선 기존 1개 -> 30개 -> 10개로 데이터를 지우지 않고 ChromaDB에 계속 집어넣으면서, DB 안에는 무려 **41개의 파편화된 매뉴얼 조각(Chunk)**이 둥둥 떠다니는 상태였습니다.
   여기서 사용자가 "T-101"을 검색했을 때, `T-101`이 포함된 뭉텅이도 당연히 DB 안에 있었습니다.

2. **근본 원인: 부족한 검색 Top-K 값 (`k=2`):**
   `agent.py` 코드 상에서 RAG 검색을 할 때, 수많은 데이터 조각 중 속도를 위해 오직 **가장 점수가 높은 상위 2개(`k=2`)의 문서만 뽑아오도록 제한**되어 있었습니다.
   만약 DB 안에 41개의 데이터가 뒤섞여 있고, 그 중 "1호기", "가스터빈", "온도" 같은 단어들이 G-101, T-104 등 여러 군데 포진해 있다면? 검색 엔진(Vector DB)이 유사도를 점수 매기다가 운 나쁘게 G-101 조각을 1위, 2위로 올려버리면, 정작 T-101 조각은 3위로 밀려나 AI의 눈(Context)에 아예 들어가지도 못하게 됩니다 (AI는 상위 2개만 보니까요).

## 🛠️ 해결 조치 (Solution)
따라서 이 문제를 근본적으로 해결하기 위해 두 가지 강력한 조치를 취했습니다. 

1. **검색 시야(Top-K) 확장:** AI가 너무 적은 조각(2개)만 보고 섣불리 판단하지 않도록, `agent.py`에서 검색해오는 문서의 개수를 `k=5`로 대폭 늘려 충분한 문서를 가져오게 했습니다.
2. **과거 DB(Vector Store) 강제 초기화 로직 추가:** 쓸데없는 쓰레기 데이터 조각들이 섞이면 유사도 점수가 튀어버리는 노이즈 현상을 막기 위해, 매뉴얼을 다시 로딩하기 전에 아예 기존 캐시(`data/chroma_db`) 폴더를 `shutil.rmtree()`로 시원하게 날려버리고 처음부터 다시 10개만 구축하도록 `rag_pipeline.py`를 수술했습니다.
